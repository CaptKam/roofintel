You are Replit Ghostwriter acting as a senior staff engineer + data engineer building a production-grade Commercial Property Contact Intelligence system.

GOAL
Build “RoofIntel Contacts” — a deterministic OSINT enrichment pipeline that, given a commercial building address (or parcel/APN), returns the best-available contact information for owners, property managers, and decision-makers, with:
- accuracy-first (provenance + confidence scoring)
- compliance-first (no doxxing, no illegal scraping, no bypassing paywalls, no credential stuffing, no ToS violations)
- auditable (source URLs + timestamps + extraction method per field)
- resilient (retries, backoff, circuit breakers, caching, queue-based jobs)
- observable (structured logs, job traces, metrics, admin debug UI)

NON-NEGOTIABLE CONSTRAINTS
1) Only use data that is publicly available AND legally/contractually permissible to access (respect robots.txt, site ToS, and rate limits). Prefer official government sources and APIs.
2) Do NOT scrape personal data from social networks (Facebook/Instagram), people search sites, or any source that appears to target private individuals. We are targeting professional/business contact points for commercial property ownership/management.
3) Output must include provenance for each data point: source name, URL (or API endpoint), time fetched, and transformation steps.
4) Every contact field must have a confidence score (0–100) and validation status.
5) Provide a “cannot verify” status rather than guessing.

TECH STACK (use these)
- Backend: Node.js + TypeScript + Express
- DB: PostgreSQL + Drizzle ORM
- Jobs: BullMQ + Redis (or a simple PostgreSQL-based queue if Redis unavailable)
- HTTP: undici or axios
- Parsing: cheerio for HTML, pdf-parse for PDFs when needed
- Frontend admin: React + Vite + Tailwind + shadcn/ui
- Deployment: Replit (env vars for secrets); write a health endpoint.

Create Drizzle schemas and migrations for:

PIPELINE (deterministic stages)
Implement a staged enrichment pipeline with idempotency:
Stage A: Normalize input + geocode (optional) + county detection.
Stage B: Parcel/assessor lookup (prefer official county appraisal district / assessor APIs when available; otherwise mark as “manual source needed”).
Stage C: Owner extraction (owner name + mailing address + entity type).
Stage D: Management company discovery:
  - from recorded property management fields if present in official sources
  - from building website if official site exists
  - from lease brochures / PDFs that are publicly posted on reputable CRE listing sites (only if ToS permits; prefer sources that provide feeds/APIs)
Stage E: Contact harvesting for orgs:
  - website domain discovery
  - parse “Contact”, “About”, “Property Management”, and tenant portal pages
  - extract business phones/emails using strict regex + context rules (avoid personal emails if context suggests private)
Stage F: Validation:
  - phone: normalize to E.164, check plausible area code, optionally use lightweight validation rules (no paid phone validation APIs unless configured)
  - email: syntax + domain MX check (optional), and match domain to org site
Stage G: Scoring + conflict resolution:
  - compute confidence from source trust, recency, corroboration count, domain match, and extraction quality
  - if conflicts, store all candidates; pick highest score if margin >= threshold, else mark unresolved.

SOURCE TRUST MODEL
Create a source reputation table (hardcoded config in /src/config/sourceTrust.ts) with base scores:
- Government assessor/recorder: 90–100
- Official org website: 75–90
- Reputable CRE brochures (public PDFs): 60–80
- Aggregators/directories: 30–60 (only if ToS permits and business-focused)
In scoring, multiply by:
- Recency factor (newer = higher)
- Corroboration factor (# independent sources matching)
- Domain match factor (email domain matches org website domain)
- Extraction factor (clean context vs footer spam)
