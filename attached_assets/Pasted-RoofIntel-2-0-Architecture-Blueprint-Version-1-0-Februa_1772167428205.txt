1. Executive Summary
RoofIntel 2.0 is a complete architectural evolution of the current production system. It preserves 100% of your existing data, agents, scoring, evidence provenance, storm monitoring, and ROI logic while eliminating the “scattered” feeling and making every new market a 1-day onboarding exercise.
Key Outcomes

New market (Houston, Austin, Chicago, etc.) = 1 day (config only).
Hail-chaser experience becomes the default login (storm dashboard first).
ROI Engine becomes the single gatekeeper for all enrichment.
Code duplication reduced by ~65%.
Future velocity 3–5× higher.
Ready for 500k+ leads and 20+ markets by end of 2026.

This is not a rewrite from scratch — it is a reorganization with dual-write migration so DFW users experience zero disruption.

2. Design Principles

Config-First Everywhere — CAD endpoints, NOAA bounding boxes, permit APIs, ROI thresholds, hail-season multipliers all live in market_data_sources + markets config.
Domain-Driven Design — 6 bounded contexts (Property, Storm, Owner, Enrichment, Scoring, Presentation). Each owns its data, events, and repository.
Event-First — All state changes publish immutable domain events (HailSwathDetected, RoiDecisionMade, LeadEnriched, etc.).
Hail-Chaser First — Default UX is storm-centric; full intelligence platform is behind “Pro Mode” toggle.
Intelligence as a Lake — leads becomes a thin hub (35 columns). Heavy data stays in normalized satellites + materialized views.
ROI as Gatekeeper — No enrichment (free or paid) runs without positive ROI decision.
Idempotent & Stateless Agents — Every agent is a pure function + event emitter.
Observability by Default — Every decision, event, and spend is auditable.

5. Data Model 2.0 (Evolution)
Keep (unchanged)

All satellite tables (contact_evidence, intelligence_claims, building_permits, portfolios, rooftop_owners, hail_events, etc.)
markets, market_data_sources, storm_runs, response_queue

Thin the leads table (from ~80→35 columns):

Core identity + FKs only.
All derived data moves to materialized views (lead_storm_view, lead_owner_view, lead_roi_view).

New Tables (Appendix A)

domain_events (event sourcing lite)
zip_tiles (pre-computed scores + cached GeoJSON)
materialized_lead_views (refreshed hourly or on event)
enrichment_budgets (per-market daily/seasonal caps)

Indexes & Performance

GIN on all JSONB.
Materialized views with concurrent refresh.
Read replicas for dashboard/map queries.


6. Domain-Driven Architecture (Bounded Contexts)
Storm Context (real-time first)

HailSwathDetected, StormThreatPredicted
Real-time Xweather + NOAA workers
Response queue with ETA calculation

Property Context

Generic importer using market_data_sources
Building footprints, permits, violations
Idempotent upsert by source_type + source_id

Owner Context

16-agent intelligence pipeline (now event-triggered)
Portfolio detection, DM inference
LLC chain resolution

Enrichment Context (see section 9)

ROI Engine (your T002)
Evidence recorder
Paid/free gating

Scoring Context

Lead scoring v3 + v4 (hail-first weights)
ZIP tiling (your T003)
ROI decisions stored per lead

Shared Context

Domain events, compliance gates, audit logging


7. Event-Driven Core & Messaging
Implementation

Redis Streams for high-throughput events
PostgreSQL LISTEN/NOTIFY for low-latency internal triggers
Each event has:
eventId, eventType, marketId, payload, timestamp, version


Key Events

LeadImported
HailSwathDetected
RoiDecisionMade
EnrichmentExecuted
LeadScoreUpdated
ZipTileUpdated

Benefits

Decoupled contexts (add Chicago without touching DFW code)
Replayable audit trail
Real-time push to Hail-Chaser map via WebSocket


8. Pipeline & Orchestration 2.0
Replace linear 9-phase orchestrator with event-driven DAG using BullMQ or Temporal.
New Flow

Import → LeadImported event
Storm context listens → tags hail
Enrichment context listens → runs ROI
Scoring context listens → updates scores + ZIP tiles
Frontend subscribes to ZipTileUpdated + LeadScoreUpdated

Admin UI

Visual DAG editor (new tab)
One-click “Force Hail Swath Processing” for testing

13. Multi-Market Scaling Strategy
Onboard New Market Steps

Admin → “Add Market” → name, center, counties, NOAA bbox
Add 4–8 rows to market_data_sources (CAD endpoint, permit API, etc.)
Run importers/run-all for that market
System auto-generates ZIP tiles + first ROI batch
Done — heatmap and Hail-Chaser ready


14. Migration Path (Zero Downtime — 4–6 Weeks)
Phase 1 (Week 1) — Dual-write new domains
Phase 2 (Week 2) — Backfill domain events from existing data
Phase 3 (Week 3) — Materialized views + read-replica cutover
Phase 4 (Week 4) — Deprecate old agents/files
Phase 5 (Week 5–6) — Ship Hail-Chaser default + new admin DAG editor
All old routes remain functional via compatibility layer.
