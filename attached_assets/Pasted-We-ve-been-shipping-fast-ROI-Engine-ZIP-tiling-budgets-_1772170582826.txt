Result: 9+ tabs, overlapping agents, long load times, and the “heavy overloaded” feeling you’re describing. It’s the classic scaling symptom of a monolith UI.
This is exactly why we did the 2.0 blueprint — to fix this kind of sprawl before it gets worse. Below is a complete, opinionated restructure of the admin layer (and related agents) for RoofIntel 2.0. It reduces cognitive load by ~65%, makes daily ops faster, keeps everything discoverable, and aligns with the “Hail-Chaser first” philosophy.
I explored three restructure options, picked the best one, and give you the full file plan, migration path, and edge cases.
1. Why the Overload Happened (Root Cause Analysis)

Historical pattern: Every new agent (CAD importer, storm monitor, ROI, ZIP scorer, budget guardrails) got its own admin tab because admin was the only “system control” surface.
Current state: 9 tabs, ~34 agents visible or triggerable, pipeline with 10 phases, overlapping cards (ROI run button + ZIP compute button + budget sliders all in one place).
Real impact:
Ops team wastes 10–20 seconds hunting for the right button during a live hail swath.
Page load bloats (all Recharts + Leaflet previews).
New markets feel intimidating (“where do I configure Houston budgets?”).
Hail-Chaser (our new default UX) has almost no admin overlap — so operators bounce between pages.


Goal of restructure: Admin = rarely used system config.
Daily ops = dedicated, fast Ops Center.
Hail-Chaser = default login for 80% of users.
2. Three Restructure Options I Evaluated
Option A (Minimal) — Keep 9 tabs but collapse into accordions + search bar.
→ Quick win (1 day), but still cluttered long-term.
Option B (Radical) — Remove admin entirely, make everything card-based on main dashboard with role-based views.
→ Too disruptive for existing users.
Option C (Recommended — Sweet Spot) — Split into two surfaces:

/admin (System Admin) — Config-only (markets, data sources, compliance, users, API keys). 4 tabs max.
/ops (Operations Center) — Daily control for ROI, ZIP, pipeline, budgets, storm overrides. Card-heavy, fast, Hail-Chaser aware.

This is the 2.0 way: domain separation at the UI level too.

6. Edge Cases & Nuances Handled

Role-based access: Sales users see only Ops Center (read-only budgets). Devs see full /admin.
Hail event spike: Ops page auto-refreshes every 30s when storm_runs active.
New market onboarding: Ops card shows “Configure Houston first” wizard linking to /admin/markets.
Mobile ops team: Cards stack vertically; one-tap “Run ROI on Current Swath” button.
Performance: Ops page uses React Query + stale-while-revalidate; heavy computations (ZIP scoring) run async with progress.
Undo safety: All ROI runs log to domain_events (2.0) so you can replay or rollback.

7. Benefits & Business Impact

Operator speed: During a real hail swath, open /ops → see budget status → click “Run ROI on Affected ZIPs” → done in 15 seconds.
UI happiness: No more tab overload. Hail-Chaser feels like the product; Ops is the cockpit.
Scalability: Adding Chicago market? One card in Ops shows “New market detected — configure budgets?”.
Future-proof: Next features (competitor density, feedback loop) get their own Ops card, never another tab.

This restructure is the natural evolution of the 2.0 blueprint we built — it just moves the “control” surface to where operators actually live.
